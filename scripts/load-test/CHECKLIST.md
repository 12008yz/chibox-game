# Пошаговая проверка масштабирования (до 500+ пользователей)

Делайте шаги по порядку. Не переходите к следующему, пока текущий не выполнен без ошибок.

---

## Часть 1. Подготовка (один раз)

### Шаг 1.1 — Установить зависимости для нагрузочных тестов

В папке **backend** выполните:

```bash
npm install
```

Убедитесь, что в `package.json` в devDependencies есть `autocannon` и `artillery`. Если только что добавили — команда выше их подтянет.

**Ожидаемый результат:** установка завершается без ошибок (exit code 0).

---

### Шаг 1.2 — Проверить, что приложение вообще запускается

В папке **backend**:

```bash
npm start
```

Подождите 5–10 секунд. В консоли должны появиться строки вроде:
- «База данных подключена» или «Соединение с базой данных успешно»
- «HTTP сервер запущен на порт 3000» (или ваш PORT)
- «Socket.IO готов.»

Если есть Redis: «Redis: сессии и rate-limit используют Redis» или «Socket.IO: Redis adapter включён».
Если Redis нет: «Сессии: PostgreSQL (Redis недоступен)» — это нормально для локалки.

**Ожидаемый результат:** сервер запустился, в логах нет красных ошибок (stack trace).  
Остановите сервер: **Ctrl+C**.

---

## Часть 2. Проверка на локальной машине

### Шаг 2.1 — Запустить сервер снова

В первом терминале:

```bash
cd backend
npm start
```

Оставьте этот терминал открытым (сервер должен работать).

---

### Шаг 2.2 — Быстрый HTTP-тест (autocannon)

Откройте **второй** терминал. В папке **backend**:

```bash
npm run load-test
```

Подождите ~25 секунд. В конце будет таблица: **requests**, **latency**, **throughput** (RPS), **errors**.

**На что смотреть:**
- **errors** — должен быть 0. Если есть ненулевые — смотрите в первом терминале (логи сервера), что за ошибки.
- **throughput** — сколько запросов в секунду. На локалке может быть сотни–тысячи; точная цифра не важна, важно что тест прошёл.

**Ожидаемый результат:** errors = 0, в логах сервера нет падений (uncaught exception, ECONNREFUSED и т.п.).

---

### Шаг 2.3 — Сценарий по нескольким эндпоинтам (Artillery)

Во втором терминале (сервер в первом по-прежнему запущен):

```bash
npx artillery run scripts/load-test/artillery-http.yml
```

Дождитесь окончания. В конце Artillery выведет отчёт: **Summary report**, **Scenarios launched**, **RPS**, **Response time (min/max/median)**.

**На что смотреть:**
- В блоке **Codes** не должно быть большого числа 5xx или 429 (допустимо немного 429, если упёрлись в rate limit).
- **Response time** — на локалке обычно низкие (десятки миллисекунд). Если median > 1000 ms — уже повод смотреть логи и БД.

**Ожидаемый результат:** тест завершился, преобладают коды 200, нет массовых 500.

Остановите сервер в первом терминале (**Ctrl+C**), если дальше будете тестировать только прод.

---

## Часть 3. Проверка на продакшене

Перед нагрузкой на прод убедитесь:
- есть бэкап БД (или вы в тестовом окружении);
- вы можете смотреть логи/метрики сервера (PM2, логи приложения, мониторинг CPU/RAM).

### Шаг 3.1 — Убедиться, что прод доступен

В браузере или в терминале:

```bash
curl -s -o Nul -w "%{http_code}" https://api.chibox-game.ru/api/v1/cases
```

Должно вернуться **200** (или 304 и т.п. — не 5xx и не таймаут).

**Ожидаемый результат:** код 200 (или другой успешный).

---

### Шаг 3.2 — Первый лёгкий нагрузочный тест на проде

В терминале (из любой папки, где есть Node/npx):

```bash
npx autocannon -c 20 -d 20 https://api.chibox-game.ru/api/v1/cases
```

- **-c 20** — 20 одновременных соединений  
- **-d 20** — длительность 20 секунд  

**На что смотреть:**
- В выводе autocannon: **errors** (должен быть 0 или очень мало).
- На сервере: логи приложения, нагрузка на CPU/RAM. Не должно быть падений процесса.

**Ожидаемый результат:** errors = 0 или единицы, сервер не падает, в логах нет критических ошибок.

---

### Шаг 3.3 — Увеличить нагрузку (средняя)

Если шаг 3.2 прошёл спокойно:

```bash
npx autocannon -c 50 -d 30 https://api.chibox-game.ru/api/v1/cases
```

Снова смотрите: **errors**, логи сервера, CPU/RAM. Если появляются **429** — срабатывает rate limit (500 req/min на IP); для одного клиента теста это нормально, при реальных 500 пользователях с разных IP лимит не так быстро достигнется.

**Ожидаемый результат:** ошибок мало или ноль, сервер стабилен.

---

### Шаг 3.4 — Сценарий Artillery на проде

Задайте базовый URL и запустите сценарий:

**Windows (cmd):**
```cmd
set BASE_URL=https://api.chibox-game.ru
npx artillery run scripts/load-test/artillery-http.yml
```

**Windows (PowerShell):**
```powershell
$env:BASE_URL="https://api.chibox-game.ru"
npx artillery run scripts/load-test/artillery-http.yml
```

**Linux/Mac:**
```bash
BASE_URL=https://api.chibox-game.ru npx artillery run scripts/load-test/artillery-http.yml
```

В отчёте смотрите:
- **Codes** — сколько 200, 429, 5xx.
- **Response time (median, p95)** — приемлемые ли задержки для вашего продукта.

**Ожидаемый результат:** большинство запросов с кодом 200, 5xx — редкость или отсутствуют.

---

### Шаг 3.5 — Ближе к 500 «пользователям» (опционально)

Если 3.3 и 3.4 прошли хорошо, можно поднять нагрузку ещё:

```bash
npx autocannon -c 100 -d 45 https://api.chibox-game.ru/api/v1/cases
```

Или в `scripts/load-test/artillery-http.yml` увеличить в `phases`:
- **arrivalRate** (например до 80–100 пользователей/сек);
- **duration** (например 90 сек).

Запустить снова с тем же `BASE_URL`.

**На что смотреть:** логи сервера, пул БД (если есть заголовки X-DB-Pool-* или отдельный мониторинг), Redis (если используется). При нехватке соединений БД или очереди в пуле появятся задержки или ошибки — по ним можно решить, увеличивать ли DB_POOL_MAX и т.д.

**Ожидаемый результат:** вы понимаете, при какой нагрузке появляются первые проблемы (если они есть), и можете скорректировать настройки (пул БД, лимиты, ресурсы сервера).

## Часть 4. Что проверить в настройках (если что-то пошло не так)

| Симптом | Что проверить |
|--------|----------------|
| Много 429 | Rate limit: 500 req/min на IP. Для теста с одной машины нормально; для прода — при необходимости ослабить или считать по user_id. |
| 5xx, таймауты | Логи приложения, размер пула БД (DB_POOL_MAX), нагрузка на CPU/RAM, сетевая доступность БД/Redis. |
| Рост latency при росте нагрузки | Пул БД (мало соединений), медленные запросы к БД, отсутствие индексов. |
| Падение процесса | Логи (uncaught exception, OOM). Проверить лимиты памяти (например PM2 max_memory_restart), утечки. |
